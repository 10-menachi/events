# ADR 002: Use UUID Primary Keys

## Status

Accepted

## Context

The platform is intended to evolve beyond a simple CRUD application.
The long-term vision includes:

- A modular architecture
- Multiple backend services (Node.js, Laravel, Go, and potentially Rust)
- Public Restful APIs
- Possible future decomposition into microservices

Major entities such as users, organisations, events, tickets, registrations,
payments and certificates may eventually be created and managed by different services.

Using sequential integer identifiers would tightly couple identifier generation to a
single database and expose predictable identifiers through public APIs.

## Decision

The platform will use UUIDs as the primary keys for all major domain entities.

This includes, but is not limited to:

- User
- Organization
- Event
- EventSession
- Ticket
- Registration
- Payment
- Certificate
- Notification

UUIDs will be generated by the application  through Prisma and stored as
the primary identifier for each entity

## Rationale

UUIDs provide globally unique identifiers that can be generated
independently without requiring coordination from the database. This
aligns with the long-term architectural goals of the project and provides
a consistent identifier strategy accross services. Additionally, UUIDs
reduce the exposure of predictable sequential identifiers
in public APIs.

## Consequences

### Advantages

- Globally Unique Identifiers
- Suitable for distributed architectures
- IDs can be generated without database coordination
- consistent identifier strategy accross services
- public APIs do not expose sequential identifiers

### Disadvantages

- Larger indexes than integer primary keys
- Slightly increased storage requirements
- less human-readable during debugging

These trade-offs are considered acceptable for this project

## Alternatives Considered

### Auto-Incrementing Integer IDs

These were considered because they are simple, efficient and easy to debug

However, they were not selected because the project's long term goals favor
globally unique identifiers and future architectural flexibility.
